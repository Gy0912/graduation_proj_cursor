# 为何原先 0.5B 模型效果很差（调试分析）

以下为本科论文中可写的「失败原因分析」，结合常见 LLM 安全微调经验：

## 1. 容量不足（欠拟合 / 无法同时学好「代码 + 安全」）

0.5B 参数量极小，对「长上下文 + 精确 API（execute/cursor）+ 安全约束」同时建模能力弱，容易退化为背诵短模板或产生不一致输出。

## 2. 数据与任务难度不匹配

若训练数据偏合成、分布单一，而评测 prompt 稍变（表名/字段/驱动），小模型泛化差，表现为「漏洞率不降」或「修复率虚高/虚低」。

## 3. 训练配置与目标不对齐

- 学习率过大：灾难性遗忘，功能分下降  
- 学习率过小：几乎不学，安全指标不动  
- `max_seq_len`、batch、LoRA rank 与数据长度不匹配时，有效梯度噪声大  

## 4. 评测与检测器的交互

规则检测器对「边界写法」敏感；小模型更可能输出怪异但可运行片段，被规则误判或漏判，导致指标波动大。

## 5. 本项目的改进方向（已在新流水线中落实）

- 换用 **2B–3B 代码模型**（如 StarCoder2-3B）提升表达能力  
- **4bit + LoRA + batch=1 + 梯度累积** 在 8GB 上可训  
- **双方法**：SFT 学安全实现 + DPO 用偏好对抑制不安全补全
